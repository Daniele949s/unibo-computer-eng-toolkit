1. Apro il file pcapng con Wireshark per analizzare il traffico ed eventualmente il contenuto 
di alcuni pacchetti

2. Ordinando in base all'ip di destinazione dei pacchetti posso notare che la subnet a cui è destinato 
l'attacco è: 127.23.3.0/24

Mentre la rete degli attaccanti è: 127.22.2.0/24

Nel file fornito ci sono diversi protocolli in gioco
	- protocollo ARP: per fare "discovery" dei mac addresses, non è indicativo di nessun attacco (in questo caso)
	- protocollo HTTP: ci sono numerose richieste HTTP get con un payload che sembrerebbe simile a quello di una
			   classica binary exploitation (molte A per rompere un buffer, 4 B per sovrascrivere indirizzo
			   di ritorno)
	- protocollo TCP e ICMP: a primo acchitto, non sembrano esserci attacchi rilevanti

3. Utilizzo Suricata per generare degli alert in base a certe condizioni di traffico. 

    - Per analizzare il payload è opportuno impostare nel file /etc/suricata/suricata.yaml il valore
    "payload printable" a "true"

    - La variabile HOME_NET definisce la rete interna protetta, cioè l'insieme degli indirizzi IP che Suricata 
    considera "di casa", utile per distinguere il traffico in ingresso da quello in uscita. Devo cambiare sul
    suricata.yaml -> HOME_NET: "[127.23.3.0/24]"

3. Scrivo una regola suricata opportuna (vedere exam.rules)

4. Per applicare la mia regola al file pcapng uso il comando

    sudo suricata -c /etc/suricata/suricata.yaml -l logs -s exam.rules -r dump-20250213.pcap -k none

I log di output verranno inseriti nella directory ./logs

4. Analizzando il file "eve.json", posso notare che il tentativo di attacco è stato rilevato con successo
